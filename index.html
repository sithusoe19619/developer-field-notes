Git vs. GitHub: What is the Difference?

Git

Git is a distributed version control system that runs entirely on your local machine. It tracks every change you make to files in a project, lets you roll back to previous states, and lets multiple versions of your project exist side-by-side as branches. Git does not require an internet connection — all of its power lives in the hidden .git/ folder inside your project.

Think of Git as the tool — like a camera that records every snapshot of your project.

GitHub

GitHub is a cloud-based hosting platform built around Git. It stores your Git repositories online so you can access them from anywhere, share them with others, and use extra features like Issues, Pull Requests, and GitHub Pages. GitHub does not replace Git — it extends it.

Think of GitHub as the cloud album — where you upload and share the snapshots Git took.

The Daily Developer Loop

Every coding session follows the same four-step rhythm:

  1. Edit Write or modify code in your working directory — the files you can see and open.
  2. Stage Run git add <file> to move changes into the staging area. This is where you choose exactly what goes into your next snapshot.
  3. Commit Run git commit -m "message" to save a permanent snapshot of everything in the staging area. A commit is like pressing Save on a level in a video game — you can always come back here.
  4. Push Run git push to upload your commits to GitHub, keeping the remote repository in sync with your local work.

Git Command Reference

Eight commands every developer uses every day:

  • git init Initializes a brand-new Git repository in the current folder. Creates the hidden .git/ directory that tracks all history.
  • git status Shows the current state of the working directory and staging area — which files are modified, staged, or untracked. Run this constantly; it is your compass.
  • git add <file> Moves a file (or changes within a file) into the staging area, queuing it for the next commit. Use git add . to stage everything at once.
  • git commit -m "message" Records a permanent snapshot of all staged changes with a descriptive message. Write messages in imperative mood: "Add navigation bar", not "Added navigation bar".
  • git log --oneline Displays a compact history of all commits — one line per commit with its short hash and message. Great for reviewing what was done and when.
  • git diff Shows the exact line-by-line differences between your working directory and the last commit (or between staged and unstaged changes). Use before every commit.
  • git checkout -b <branch-name> Creates a new branch and switches to it in one command. Branches let you work on a feature in isolation without affecting main.
  • git push -u origin <branch> Pushes a local branch to GitHub for the first time. The -u flag sets the upstream tracking link so future git push commands know where to send commits.

Things I Should Not Commit

Some files must never enter version control. Use .gitignore to exclude them.

  • .env Contains environment variables — API keys, database passwords, secret tokens. Committing this exposes credentials to anyone who can see the repository.
  • Secrets and API keys Any hardcoded passwords, private keys, OAuth tokens, or service credentials. Once committed, they live in the git history forever — even if you delete the file later.
  • node_modules/ The folder where npm installs packages. It can contain thousands of files and hundreds of megabytes. Anyone who clones the repo can regenerate it with npm install.
  • .DS_Store A hidden macOS metadata file created automatically in every folder. It is machine-specific and meaningless to collaborators or the project.
